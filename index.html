[ how this works & security notes ]

Transport: files move over WebRTC data channels directly between your browser and your recipient's. Google public STUN servers help both sides discover their public addresses; no file data ever flows through them. If a direct connection can't be established (strict NAT), the transfer falls back to a free public TURN relay (openrelay.metered.ca) — still end-to-end encrypted, but traffic hops through the relay.

Encryption: WebRTC data channels always use DTLS-SRTP (mandatory AES-GCM). If you add a passphrase, a second layer is applied: PBKDF2-SHA256 (210k iterations) derives an AES-GCM-256 key, and every chunk is encrypted with a fresh random IV before leaving your machine. Defense in depth.

Throughput: transfers use 4 parallel data channels with 256 KB chunks and buffer-aware backpressure. Realistic ceiling is 500 Mbps–1.5 Gbps on good hardware between two LAN-adjacent machines; your ISP is usually the real bottleneck.

Receiver storage: where the browser supports the File System Access API (Chrome/Edge/Opera desktop), chunks stream straight to disk — no RAM cap, multi-GB files fine. Elsewhere, files accumulate in memory then download via blob URL (~2 GB practical cap).

No server: this page is static. Host it anywhere — GitHub Pages, Netlify, your own box, or just open the HTML file locally. The only network dependencies are Google STUN, the TURN fallback, and CDN font/library loads (which you can vendor locally if you want zero third-party requests).
